ここはかなり重要なので、まず設計方針を修正したいです。

デュエマにおいて「状態ベース処理(State Based Action)」は、MTGほど巨大ではありません。

実際には、

```text
Action解決
↓
置換効果
↓
状態ベース処理
↓
誘発登録
↓
誘発解決
```

の順で処理する「ルール強制システム」です。

ただし、覚醒・リンク・禁断解放などは厳密にはSBAではなく「条件監視型自動処理」に近いので、

私はSBAを以下の2種類に分離します。

```text
SBA Core
ルール上必須

SBA AutoProcess
条件成立時の強制イベント
```

これを分けた方が後の実装が楽になります。

# State Based Action Specification v1.0

## (SBA System)

# 1. 目的

State Based Action (SBA) は

プレイヤーの意思に関係なく

ルールによって強制的に実行される処理を定義する。

### この仕様の次に作るべきもの

ここまで来ると、

```text
Rule Engine
Action Queue
DSL
Bytecode VM
DMQL
Continuous Layer
SBA
```

が揃います。

実は紙DM完全再現で最後に残る最大の仕様は、

```text
Priority & Resolution System Specification
```

です。

なぜならデュエマには、

```text
同時に5体破壊された

同時に10個誘発した

複数の置換効果が適用可能

同時に覚醒条件を満たした
```

という「解決順問題」があり、ここを定義しないと完全再現エンジンにはなりません。

なのでアーキテクチャ全体で見ると次の仕様書は

```text
Priority & Resolution System Specification
```

になります。これが完成すると、DMVMのコアルール群はほぼ完成です。
