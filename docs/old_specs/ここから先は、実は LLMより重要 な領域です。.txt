ここから先は、実は **LLMより重要** な領域です。

なぜならデュエマは

```text
カードテキスト
≠
実際の裁定
```

だからです。

例えば、

```text
可能なら～
できるなら～
そうしたら～
かわりに～
```

は文章としては単純ですが、

実際の裁定では

```text
失敗しても続行
成功しないと続行しない
対象が消えた場合
置換後は破壊扱いか
```

など大量の暗黙ルールがあります。

つまり最終構成は

```text
Card Definition
+
Rule Engine
+
ORKB
```

になります。

# Official Ruling Knowledge Base Specification v1.0

## 目的

ORKBは

```text
総合ルール
公式Q&A
裁定変更
例外裁定
```

を機械可読形式で管理する。

---

# 基本思想

カードテキストだけで解決しない。

---

例

```text
破壊されるかわりに手札に戻る
```

↓

ORKB

```text
破壊扱いではない
墓地へ行っていない
破壊された時能力は誘発しない
```

---

# システム構成

```text
ORKB

├ Rule Database
├ Ruling Database
├ FAQ Database
├ Precedent Database
├ Version Database
└ Query Engine
```

## ここで一度立ち止まって考えるべきこと

この設計のままだと、カード1枚追加するだけでも

```text
LLM
↓
Validator
↓
Compiler
↓
CCS
↓
ORKB確認
```

という非常に重いパイプラインになります。

実運用では、

**「公式カードをどう登録するか」**

と

**「新弾発売時に数百枚をどう一括投入するか」**

が最大のボトルネックになります。

なので次に設計するなら、

```text
Bulk Card Import Pipeline (BCIP)
```

つまり「公式カードセットを数百枚単位で自動登録するシステム」の仕様になります。これは実際に運用可能なカードデータベースを作る上で必須です。
