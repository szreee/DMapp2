これは本当に重要です。

実はここまで作った

```text
DMVM
DMQL
Effect DSL
Rule Engine
```

よりも難しいのが Continuous Effect（常在効果）です。

なぜならデュエマには、

```text
+2000される
↓
さらに+3000される
↓
パワーを倍にする
↓
能力を失う
↓
種族を追加する
↓
文明を失う
```

が同時に存在するからです。

例えば

```text
元のパワー 3000

A効果
+2000

B効果
パワーを2倍

結果は？

(3000+2000)×2 =10000

なのか

3000×2+2000 =8000

なのか
```

を決めなければなりません。

---

# 設計方針

MTGほど複雑なレイヤーは不要です。

しかし

```text
能力追加
能力削除
文明変更
種族変更
コスト変更
パワー変更
```

は順序を固定する必要があります。

---

## 次の段階

ここまで作ると、エンジンの主要構成要素はほぼ揃います。

```text
Rule Engine
Action Queue
Effect DSL
Bytecode VM
DMQL
Continuous Layer
```

ただし、紙DM完全再現にはまだ最後の大物が残っています。

それが

```text
State Based Action Specification
```

です。

例えば

```text
パワー0
↓
破壊

封印0
↓
禁断解放

覚醒条件達成
↓
覚醒

リンク条件達成
↓
リンク
```

のような「ルールによる強制処理」を完全定義する必要があります。

実際の実装難易度でいうと、

```text
Effect DSL
↓
DMQL
↓
State Based Action
```

の順で難しいので、完全版を目指すなら次はSBA仕様書を作る段階になります。
