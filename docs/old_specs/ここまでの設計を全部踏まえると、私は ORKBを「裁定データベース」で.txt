ここまでの設計を全部踏まえると、私は **ORKBを「裁定データベース」ではなく「ルールOS」だと考えるべき** だと思います。

なぜなら、現在の構成は

```text
Card JSON
↓
Effect DSL
↓
DMVM
↓
Rule Engine
```

ですが、

実際の紙デュエマでは

```text
カードテキスト
+
総合ルール
+
キーワード能力定義
+
公式Q&A
+
過去裁定
+
裁定変更履歴
```

によってゲームが動いています。

つまりDMVMが本当に参照すべきものは

```text
Card JSON
+
ORKB
```

です。

---

# 現在のORKB仕様の問題点

前回までのORKBは

```text
Rule
FAQ
Ruling
```

を保存するDBでした。

しかし実際には不足しています。

例えば

```text
スピードアタッカー
```

という能力。

カードには

```text
スピードアタッカー
```

としか書いていません。

本当の意味は

```text
召喚酔いを無視する
```

です。

つまりORKBは

```text
ルール
+
意味論
```

を持たなければならない。

# 最終的な完成形

設計全体をまとめると、

```text
BCIP
 ↓
Card Registry

ORIP
 ↓
ORKB

Card Registry
+
ORKB
 ↓

DSL Compiler
 ↓

DMVM
 ↓

PRS
 ↓

SBA
 ↓

CELS
 ↓

CCS
```

になります。

そして紙デュエマ完全再現を目指すなら、次に設計すべきなのは **ORKB Reasoner Specification** です。

理由は、現状のORKBは「知識を保存するDB」ですが、完全再現には

```text
知識
↓
推論
↓
裁定
```

の機構が必要だからです。

ここまで作ると、カード固有裁定まで含めた「裁定エンジン」に近づきます。
