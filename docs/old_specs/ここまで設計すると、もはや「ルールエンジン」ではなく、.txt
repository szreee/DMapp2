ここまで設計すると、もはや「ルールエンジン」ではなく、

Duel Masters Virtual Machine (DMVM)

になります。

そして重要なのは、先ほど挙げた命令群をそのまま追加するのではなく、

カード固有ギミック
↓
汎用命令へ分解

できるように設計することです。

例えば

GR_SUMMON

を命令にすると将来の新ギミックで破綻します。

代わりに

ZONE_QUERY
RANDOM_SELECT
MOVE_TO_BATTLEZONE

の組み合わせで実現できるなら、その方が強いです。

ただしデュエマは特殊ギミックが多すぎるので、

汎用命令
+
DM専用高レベル命令

の二層構造が現実的です。

この仕様の次に必要なもの

ここまで来ると、エンジン設計の最大の難所は別の場所になります。

それは 「Query Language v1.0」 です。

例えばデュエマには、

名前に「ボルシャック」を含む

コスト5以下

ドラゴンを持つ

進化ではない

このターン出た

最後に破壊された

自分の火のクリーチャー

のような条件が大量にあります。

つまり次に作るべき仕様書は

DM Query Language Specification v1.0

です。

実際にはDSLよりも、このクエリ言語が完成した時点で「ほぼ全カードが表現可能」になります。
