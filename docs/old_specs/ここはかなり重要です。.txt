# なぜRIRが必要なのか

例えば公式裁定に

```text
「破壊されるかわりに手札に戻る場合、
そのクリーチャーは破壊されたことになりません。」
```

と書かれている。

人間には分かります。

しかしエンジンに必要なのは

```text
DestroyAttempt
↓
ReplacementApplied
↓
NotDestroyed
↓
NoOnDestroyTrigger
```

です。

つまり

```text
自然言語
↓
意味表現
↓
実行ルール
```

の中間層が必要になります。

これがRIRです。

# 私ならさらに追加するもの

実はRIRだけだとまだ弱いです。

完全再現を目指すなら、

```text
RIR
↓
RIR Graph
```

を持ちます。

---

例えば

```text
DestroyReplacement
```

は

```text
DestroyAttempt
↓
Move
↓
NotDestroyed
↓
NoDestroyTrigger
```

という依存グラフになる。

---

これを保持すると、

```text
ExplainDecision()
```

で

```text
なぜそうなったか
```

を完全に説明できます。

---

## 私が最終的に採用するRIR

もし本当に22,528枚＋全裁定を目指すなら、

```text
RIR Node
+
RIR Graph
+
Predicate System
+
Priority System
+
Conflict System
```

の4本柱にします。

これは実質的に

```text
デュエマ専用 Prolog
```

に近い構造になります。

そして次の段階では、

**Predicate Catalog Specification**

（RIRやDMQLやDSLやReasonerが共通利用する全述語の定義書）

を作るべきです。

実はこれが無いと、各システムで `Destroy` や `Move` の意味が微妙にズレ始めて、後で大規模な破綻が起きます。
