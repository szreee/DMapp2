ここまで設計が進むと、実は **BCIPよりORIPの方が重要** になります。

理由は単純で、

```text
カードテキストだけでは再現できない
```

からです。

デュエマには大量の

* 総合ルール
* 公式Q&A
* 裁定変更
* 特例裁定
* キーワード能力解釈

が存在します。

例えば、

```text
「そうしたら」
```

はカードテキスト上は4文字ですが、

実際には

```text
前の処理が成功した場合のみ実行
```

というルール知識が必要です。

これはカードから取得できません。

そのためDMVM完成形では

```text
Card Registry
+
ORKB
```

が両輪になります。

---

# Official Ruling Import Pipeline Specification v1.0 (ORIP)

## 1. 目的

公式裁定情報を収集し、

```text
機械可読なORKB
```

へ変換する。

---

# 実際の開発順序

ここで現実的な話をすると、

ORIPを最初からフル自動にするのはおすすめしません。

まずは

```text
Phase1
総合ルール

↓

Phase2
キーワード能力

↓

Phase3
重要FAQ

↓

Phase4
カード固有裁定
```

の順がいいです。

特にデュエマでは、

```text
スピードアタッカー
ブロッカー
マッハファイター
ジャストダイバー
革命チェンジ
侵略
S・トリガー
ニンジャ・ストライク
```

のようなキーワード能力だけでも数百種類あります。

なので最初のORKBは、

```text
Core Rules + Keyword Ability Database
```

として構築し、その後にカード固有裁定を積み上げる方式が、実装コストと完成度のバランスが最も良いと思います。
