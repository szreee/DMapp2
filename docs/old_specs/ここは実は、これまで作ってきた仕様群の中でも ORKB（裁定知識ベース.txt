ここは実は、これまで作ってきた仕様群の中でも **ORKB（裁定知識ベース）とDMVMを接続する最後の中核層** になります。

なぜなら、

```text
AGS
→ 構文を理解する

SBS
→ 参照先を理解する

ATC
→ テンプレートを理解する

KAFS
→ キーワードを理解する
```

まではできても、

```text
その能力は本当に裁定上同じ意味なのか？
```

はまだ分からないからです。

例えばデュエマでは、

```text
カードを引く

山札の上から1枚を手札に加える
```

は実質的に同じ処理です。

一方、

```text
破壊する

墓地に置く
```

は似ているようで全く別です。

ERCSは、

**能力の「表現」ではなく「意味」を比較するシステム**

になります。

---

# Effect Equivalence & Ruling Consistency System (ERCS) v1.0

## 目的

ERCSは

```text
能力A
能力B
```

が

```text
同じ意味か

部分的に同じか

完全に異なるか
```

を判定する。

また

```text
公式裁定
↓
DSL
↓
Bytecode
```

の整合性を検証する。

---


# Semantic Effect Graph (SEG)

能力を意味グラフへ変換する。

---

例

```text
カードを1枚引く
```

↓

```text
Draw
 └ Count=1
```

---

例

```text
山札の上から1枚を手札に加える
```

↓

```text
Move
 ├ Source=DeckTop
 ├ Destination=Hand
 └ Count=1
```

---

ERCSは

```text
Draw
```

と

```text
Move(DeckTop→Hand)
```

が等価であることを知っている。

# ERCS v2で追加すべきもの

ここまで来ると、DMVMはかなり完成しています。

ただし、本当に紙デュエマ完全再現を目指すなら、最後に必要なのは

# Comprehensive Rules Formalization System (CRFS)

です。

理由は単純で、

今までは

```text
カード能力
```

を形式化していました。

しかし実際のデュエマには

* ターン進行
* 攻撃ルール
* シールドブレイク手順
* ブロック手順
* 勝敗判定
* 山札切れ
* 同時発生処理
* 優先順位

など、

**カードに書かれていないゲームルールそのもの**

が存在します。

本当の意味での「紙デュエマ完全再現」を目指すなら、

最後は総合ルールそのものを形式言語化する

**CRFS (Comprehensive Rules Formalization System)**

が必要になります。

ここまで到達すると、DMVMは「ゲームエンジン」ではなく、「デュエマ総合ルールの実行系」になります。
