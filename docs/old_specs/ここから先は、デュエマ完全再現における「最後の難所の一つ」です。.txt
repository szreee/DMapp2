ここから先は、デュエマ完全再現における「最後の難所の一つ」です。

実際、紙デュエマの裁定事故のかなりの割合は

```text
そのクリーチャー
そのカード
こうして出したクリーチャー
このターン離れたクリーチャー
相手が選んだクリーチャー
```

などの**参照先が何なのか**に起因します。

---

# RRSの目的

RRSは

```text
カードテキスト
↓
DSL
↓
DMVM
```

の間で

```text
何を指しているのか
```

を統一的に管理する。

---

# なぜ必要か

例えば

```text
相手のクリーチャーを1体破壊する。
その後、そのクリーチャーと同じ名前を持つクリーチャーをすべて破壊する。
```

この

```text
そのクリーチャー
```

は何か？

---

人間なら

```text
最初に破壊したクリーチャー
```

と分かる。

しかしDMVMは

```text
ObjectID？
CardUID？
LKI？
```

を明確にしなければならない。

# Reference Model

RRSでは参照を5種類に分ける。

---

## Type 1 Object Reference

現在存在するゲームオブジェクト。

例

```text
このクリーチャー
この呪文
この進化クリーチャー
```

---

内部

```cpp
ObjectRef
{
    ObjectID
}
```

---

例

```text
このクリーチャーはパワー+2000
```

---

対象。

```cpp
ObjectID=57
```

---

# Type 2 Card Reference

カード実体。

---

例

```text
そのカードを手札に戻す
```

---

内部。

```cpp
CardRef
{
    CardUID
}
```

---

OICS連携。

---

ゾーン移動後も追跡可能。

---

# Type 3 LKI Reference

Last Known Information。

---

例

```text
破壊されたクリーチャーのパワーを参照する
```

---

離脱済み。

---

Objectは消滅。

---

LKIを参照。

```cpp
LKIRef
{
    SnapshotID
}
```

---

# Type 4 Group Reference

集合。

---

例

```text
これらのクリーチャー
```

---

```cpp
GroupRef
{
    ObjectIDs[]
}
```

---

# Type 5 Query Reference

動的参照。

---

例

```text
相手のクリーチャー
```

---

実行時検索。

```cpp
QueryRef
{
   DMQLQuery
}
```

# 現時点で不足しているもの

ここまでで

* DMQL
* DSL
* Bytecode
* PCS
* ETS
* ZTS
* OICS
* RRS

が揃いました。

しかし、デュエマ完全再現にはもう一つ巨大な仕様が必要です。

それが

```text
Effect Duration & Dependency Specification (EDDS)
```

です。

なぜかというと、

```text
ターンの終わりまで

次の相手ターンのはじめまで

このクリーチャーが離れるまで

進化元があるかぎり
```

のような効果持続時間や依存関係が大量に存在するからです。

実際、継続効果レイヤー（CELS）を本当に完成させるには、EDDSが必要になります。これはデュエマ再現における「継続効果の憲法」に相当します。
