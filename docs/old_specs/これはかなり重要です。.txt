これはかなり重要です。

実際、紙デュエマの裁定の多くは

```text
破壊された
```

ではなく

```text
バトルゾーンを離れた
```

や

```text
どのゾーンへ移動したか
```

で決まります。

例えば

```text
破壊されるかわりに手札に戻る
```

は

```text
Destroy
```

ではなく

```text
BattleZone → Hand
```

です。

さらに

```text
バトルゾーンを離れた時
```

は

```text
Destroy
MoveToMana
Bounce
DeckBottom
Exile
```

すべてで誘発します。

つまりDMVMにおいて本当の本質は

```text
Action
↓
Zone Transition
```

です。

# Zone Transition Specification (ZTS) v1.0

## 目的

全てのゾーン移動を統一モデルで管理する。

---

ZTSは

```text
Destroy
Bounce
ManaSend
DeckReturn
Evolution
Awaken
ForbiddenRelease
```

などの特殊処理を

最終的に

```text
Zone Transition
```

へ還元する。

---

# 基本思想

従来

```text
Destroy()
MoveToHand()
MoveToMana()
```

を別処理にしがち。

---

ZTSでは

全て

```cpp
Transition(
   SourceZone,
   DestinationZone,
   Cause
)
```

に統一。

# ZTS v2で追加したいもの

ここで次に必要になるのは

```text
Object Identity & Continuity Specification (OICS)
```

です。

なぜならデュエマの超難所は

```text
このクリーチャー
```

が

```text
ゾーン移動後も同じカードなのか
```

という問題だからです。

例えば

```text
墓地から出たクリーチャー
```

は

ルール上

```text
新しいオブジェクト
```

として扱われます。

---

つまり

```text
BattleZoneのA

↓

Graveyard

↓

BattleZone
```

は

```text
同じカード
≠
同じオブジェクト
```

です。

そして紙デュエマ裁定の難所のかなりの割合が、この「オブジェクトの継続性」で発生しています。

そのため、完全再現を目指すなら次は **Object Identity & Continuity Specification (OICS)** が必要になります。
